----------------------------------------------------------------------------------## Steps to Automated Bug Fixing AI ##-------------------------------------------------------------------------------------

Step 1	Monitor	
Action: The AI ("Sentinel") constantly pings the website every second to check its speed.
Reason: Problems happen instantly. If we wait for a user to complain, we are too late. We need real-time awareness.

Step 2	Detect	
Action: The AI calculates the Z-Score. If the latency jumps from 30ms to 1500ms, it flags an "Anomaly."
Reason: We use math (Z-Score) so the AI ignores small hiccups but catches real disasters.

Step 3	Search	
Action: The AI ("Librarian") converts the error ("High Latency") into math vectors and searches your codebase for related words like sleep, wait, or timeout.
Reason: The codebase might be 100 files long. The AI needs to find the exact file causing the problem without reading everything manually.

Step 4	Reason	
Action: The AI ("Architect" - LLM) looks at the code found in Step 3. It sees time.sleep inside an async function and thinks: "Aha! This is a blocking call. That's the bug."
Reason: This requires intelligence. A simple script can't understand why code is bad. An LLM can understand context.

Step 5	Patch	
Action: The AI writes a new version of the code where time.sleep is replaced with await asyncio.sleep.
Reason: Identifying the bug isn't enough; we need to actually fix it.

Step 6	Verify	
Action: The AI restarts the server with the new code and checks the speed again.
Reason: Trust but Verify. AI sometimes makes mistakes. We never deploy a fix unless we prove it actually works.

------------------------------------------------------------------------------------## chaos_service.py ##--------------------------------------------------------------------------------


What does chaos_service.py do?

This AI is used for Automated Site Reliability Engineering (SRE). 
This script creates a web server that intentionally has a split personality.

* Normally: It is fast and efficient, capable of handling thousands of users at once.
            "Handling" means: Responding to Requests (API Calls)
            When we say "handling thousands of users," we don't mean users logging into a website with a username and password. We mean thousands of computers asking your server a question at the same exact second.

In the software world, this is called "High Throughput" or "Concurrency."

* When Sabotaged: It switches to "bad code" (a blocking sleep timer) that freezes the entire server.
                  "Sabotage" is a simulation

In simple terms: It is a "Digital Mechanic" that lives inside your server.
Without this AI: When a website gets slow, a human engineer has to read logs, find the bad code, write a fix, test it, and deploy it. This takes hours.
With this AI: The AI detects the slowness, finds the bad code, fixes it, and deploys the solution automatically in seconds.

In short: It is a simulator that lets you break a server on purpose so you can test if your AI Agent is smart enough to fix it.

---------------------------------------------DRY RUNNING THE CODE (WHAT HAPPENS STEP-BY-STEP)-----------------------------------------#

Here is exactly what happens inside the computer when this code runs.

A. Initialization Phase (Lines 1-15)
Line 2-5 (Imports): Python loads the tools. FastAPI (web server), asyncio (good waiting), time (bad waiting), random (math).
Line 8 (app = FastAPI()): The computer creates an empty web server object. It is now listening for instructions but isn't running yet.
Line 15 (system_state = ...):
The computer creates a dictionary (a global variable) in memory: {'status': 'HEALTHY', 'degradation_factor': 0.0}.
Current Status: The system thinks it is healthy.

B. The Main Route: /health (Lines 20-52)
Line 20 (@app.get("/health")): The server registers this URL. If a user visits /health, the function below triggers.
Line 21 (async def health_check():): The function starts. async tells Python, "I might pause during this function, but don't freeze the whole computer while I wait."

Scenario 1: The System is Healthy (Default)
Line 24 (base_latency = ...): The computer picks a random number, say 0.03 (30 milliseconds). This simulates how long a fast database query takes.
Line 27 (if system_state["status"] == "DEGRADED":):
The computer checks the global variable.
It sees status is "HEALTHY".
Result: It skips the entire if block (lines 28-43).
Line 49 (await asyncio.sleep(base_latency)):
CRITICAL: The function pauses for 0.03 seconds.
The Magic: Because we used await, the server releases the CPU. If 100 other users try to connect during this 0.03s, the server can accept them immediately.
Line 52 (return ...): The wait is over. The server sends back: {"status": "healthy", "latency_ms": 30.0}.

Scenario 2: The System is Sabotaged (After calling /sabotage)
(Assume state is now "DEGRADED")
Line 24: Picks random number: 0.03.
Line 27 (if ... == "DEGRADED":): The condition is True. It enters the block.
Line 30 (added_latency = ...):
Math: 1.0 (factor) * 1.2 (random). Total extra delay is 1.2 seconds.
Line 37 (time.sleep(base_latency + added_latency)):
CRITICAL FAILURE: The computer runs time.sleep(1.23).
The Disaster: This function does not have await. The Python process completely stops.
If 100 other users try to connect now, they are blocked. Their browsers will spin and load nothing until this 1.23 seconds is finished.
Line 40 (return ...): The freeze ends. The server sends back: {"status": "struggling", "latency_ms": 1230.0}.

C. The Sabotage Switch: /sabotage (Lines 57-64)
Line 57 (@app.post("/sabotage")): Registers a "POST" route. You cannot visit this in a browser URL bar (which is GET); you must send a command (like clicking a button in docs).
Line 58: Function sabotage_system starts.
Line 60 (system_state["status"] = "DEGRADED"): The global memory is updated. The flag is raised.
Line 62 (system_state["degradation_factor"] = 1.0): Sets the penalty to maximum.
Line 64: Returns {"message": "SYSTEM COMPROMISED..."}.

D. The Fix Switch: /fix (Lines 68-75)
Line 68: Registers route /fix.
Line 71-73: Resets the global memory back to "HEALTHY" and 0.0.
Line 75: Returns {"message": "SYSTEM RESTORED..."}.

E. The Engine Start (Lines 79-85)
Line 79 (if __name__ == "__main__":): Checks if you are running this file directly. (True).
Line 81 (import uvicorn): Loads the server engine.
Line 85 (uvicorn.run(...)): Starts the server on port 8000.
                            This line loops forever, waiting for requests. It only stops if you hit Ctrl+C.


1. FastAPI (FastAPI): 
A modern framework for building APIs. It's used here to create the server that will run our app.

2. Decorators (@app.get): 
These tell the server, "When someone goes to this URL, run this specific function."

3. Type Hinting (: str, : float): 
We explicitly tell Python what data type to expect. This reduces bugs.

4. Async/Await: This is the most critical part.
async def: Defines a function that can pause to let other things happen.
await: The command to pause and wait for something (like a timer or database) without freezing the whole server.

5. The Bug (time.sleep vs asyncio.sleep):
asyncio.sleep(1): "I'm going to nap for 1 second. You guys keep working." (Good).
time.sleep(1): "STOP EVERYTHING! I am sleeping for 1 second. Nobody move!" (Bad - this is the bug we are creating).

---------------------------------------------------------------------------------## Dockerfile ##--------------------------------------------------------------------------------------------------------

What does the Dockerfile do?

A blueprint for building a mini-computer (Container) that runs your code. It is like a recipe that says:
1. Start with a clean Linux system (python:3.12-slim).
2. Copy your code into this system.
3. Install the libraries your code needs (from requirements.txt).
4. When the container starts, run this command (uvicorn chaos_service:app --host 0.0.0.0 --port 8000).

Without Docker: 
You send me your code. I try to run it, but I have a different version of Python, or Windows instead of Mac. It crashes. This is the "It works on my machine!" problem.

With Docker: 
You wrap your code + Python + all libraries into a Container. I run the container, and it works exactly the same on my machine as it does on yours.

Why we need it for Aethelgard: 
Your AI agent is going to be powerful. It will have permission to delete files and kill processes.

If you run this on your main Windows, the AI might accidentally delete your system files.
If you run it in Docker, the AI is locked inside a box. If it destroys the box, you just throw the box away and get a new one. Your Windows stays safe.

For the command line after making the docker file: docker build -t aethelgard-chaos .
docker build: "Hey Docker, read the Dockerfile."
-t aethelgard-chaos: "Tag (name) this image 'aethelgard-chaos'."
.: "Look for the Dockerfile in the current directory."
